# RepoCtx Guard — 开发准备文档

> 版本：v1.0-draft  
> 性质：产品定义 + 技术规范 + 数据结构规范  
> 约束：本文档只包含定义、规范与约束，不包含任何可执行代码。

---

## 1. 项目定位与核心目标

### 1.1 产品定位

**RepoCtx Guard**（内部代号），全称 **AI-assisted Development Control Plane**，中文定义为：

> 面向复杂研发项目的 AI 辅助开发控制层。

它既不是 coding agent，也不是 IDE 插件。它是一个站在 AI coder 旁边的工程控制系统，职责是对 coder 进行约束、提醒、审查与记录。

### 1.2 核心目标

用一句话概括：

> 让 AI coder 在复杂项目和长周期实验中，不要乱改、不要乱飘、不要忘上下文、不要破坏老代码、不要让新代码变成未来屎山。

### 1.3 工程宪法（Engineering Constitution）

系统所有行为必须服从以下七条宪法原则，不可违背：

| 编号 | 原则（英文） | 原则（中文） |
|------|-------------|-------------|
| EC-1 | AI must not work without project context. | AI 不能在不知道项目上下文的情况下乱改。 |
| EC-2 | New code must not become future legacy debt. | 新代码不能从诞生开始就变成未来屎山。 |
| EC-3 | Every behavior change needs protection. | 任何行为变化都必须有测试、影响分析或明确确认。 |
| EC-4 | Legacy core is production asset, not editable code. | 老项目底层代码是生产资产，不是普通可编辑实现。 |
| EC-5 | Reuse public surfaces, not internal guts. | 复用已有公开能力，不要为了局部需求修改底层实现。 |
| EC-6 | Large changes must be split. | 改动过大必须提醒 commit / split。 |
| EC-7 | Unexpected experimental results must not be automatically rationalized. | 实验结果违反预期时，必须同时怀疑理论和怀疑实现。 |

## 2. 术语表与精确定义

本文档中所有术语按字母序排列，凡后文出现的术语必须与此处定义一致，不得产生二义性。

### 2.1 通用术语

| 术语 | 英文全称 | 定义 |
|------|---------|------|
| **AI Coder** | AI Coding Agent | 指实际执行代码修改任务的 AI 实体，例如 Cursor Agent、GitHub Copilot Chat、Claude Code 等。RepoCtx Guard 不替代它，而是约束它。 |
| **变更集** | Change Set | 当前 working tree 中所有新增、修改、删除的文件集合，以 Git diff 的口径为准。 |
| **上下文** | Context | 发给 AI Coder 的、与当前任务直接相关的项目信息，理想长度控制在 1500–3000 token 之间，必须是最小但足够准确的。 |
| **受保护核心** | Protected Core | 被标记为不可随意修改的底层代码资产，包括但不限于核心服务、公共工具、认证模块、计费模块等。修改受保护核心需要走特殊审批流程。 |
| **可复用能力** | Reusable Capability | 项目中已经存在的、可通过公开接口调用的稳定功能。系统必须引导 AI Coder 复用这些能力，而非重新实现。 |
| **模块** | Module | 项目中具有独立业务语义的功能单元，可对应一个 Django app、一个前端页面目录、一个微服务或一个独立的业务域。 |
| **工程宪法** | Engineering Constitution | 见 1.3 节，七条不可违背的开发原则。 |
| **知识图谱** | Repo Knowledge Graph | 以图结构描述项目实体（文件、模块、函数、类、API）及其关系（归属、调用、依赖、数据流）的索引体系。 |

### 2.2 功能相关术语

| 术语 | 定义 |
|------|------|
| **Context Router** | 功能 1 的核心机制。不是生成一份超长文档，而是根据用户输入的任务描述，自动路由到相关模块、文件、风险点和可复用能力，并生成最小上下文。 |
| **Structure Guard** | 功能 2 的核心机制。持续检查新代码是否符合架构规则、文件结构规范和开发原则，对新代码进行实时塑形。 |
| **Regression Guard** | 功能 3 的核心机制。分析变更对测试的影响，识别测试缺口，要求行为变化必须有测试保护。 |
| **Legacy Core Guardian** | 功能 3.5 的核心机制。保护老代码底层资产不被随意修改，同时维护可复用能力索引。 |
| **双轨诊断** | Dual-Track Diagnosis，功能 4 的核心机制。实验结果异常时，必须同时从两条路径分析：Track A（科学解释，假设代码正确）和 Track B（实现怀疑，假设理论正确）。 |

### 2.3 实验相关术语

| 术语 | 定义 |
|------|------|
| **实验** | Experiment | 具有明确输入、命令、环境配置和预期输出的长周期计算任务，例如模型训练、存算一体仿真、NDN 实验等。 |
| **实验记忆** | Experiment Memory | 每次实验运行后自动沉淀的结构化记录，包含配置、结果、异常、诊断和建议。 |
| **预检** | Preflight Check | 实验运行前的环境、依赖、配置正确性检查。 |
| **运行时监控** | Runtime Monitor | 实验运行期间的存活检测、资源监控和超时告警。 |
| **运行后总结** | Post-Run Summary | 实验结束后自动生成的结构化摘要，包含指标变化、异常识别、双轨诊断和下一步建议。 |

## 3. 系统架构与模块关系

### 3.1 高层架构

系统由五个功能模块组成，按优先级分为：

1. **功能 1：Repo Knowledge Graph / Context Router**（地基，所有其他模块的数据源）
2. **功能 2：Engineering Principle / Structure Guard**（新代码约束）
3. **功能 3：Test Impact & Regression Guard**（行为保护）
4. **功能 3.5：Legacy Core Guardian**（老代码资产保护）
5. **功能 4：Experiment Intelligence Agent**（实验管理，独立扩展模块）

### 3.2 模块间数据流与闭环

标准研发任务的工作流闭环如下：

```
用户提出任务
    ↓
[功能 1] 生成任务上下文（模块、文件、风险、可复用能力）
    ↓
AI Coder 开始修改代码
    ↓
[功能 2] 持续检查新代码结构和原则合规性
    ↓
[功能 3] 检查测试覆盖和回归风险
    ↓
[功能 3.5] 检查是否触碰受保护核心
    ↓
Commit 前统一 Gate（功能 2 + 3 + 3.5 的汇总检查）
    ↓
如果是实验任务
    ↓
[功能 4] 接管运行、监控、总结、双轨诊断、通知
```

### 3.3 模块间依赖关系

- **功能 1 是唯一定向数据源**：功能 2、3、3.5 都依赖功能 1 生成的知识图谱和上下文。
- **功能 3.5 与功能 1 的双向关系**：功能 1 在生成上下文时需要引用功能 3.5 维护的受保护核心索引和可复用能力索引。
- **功能 4 相对独立**：功能 4 主要管理实验生命周期，但在实验涉及代码修改时，仍需经过功能 2、3、3.5 的 Gate。
- **Commit Gate 是功能 2、3、3.5 的聚合输出点**：Commit Gate 不独立产生新规则，只汇总三个模块的检查结果。

## 4. 功能模块详细定义

以下每个功能模块的定义包含：输入、输出、核心机制、数据结构要求和边界情况。

### 4.1 功能 1：Repo Knowledge Graph / Context Router

#### 4.1.1 目的
解决 AI Coder 在新对话中不了解项目结构、读错文件、漏掉关键模块、误解业务流的问题。

#### 4.1.2 输入
- 用户以自然语言描述的任务，例如 `"我要改 free call 登录流程"`。
- 当前项目的完整代码库文件树（通过扫描获得）。
- 预先定义或自动生成的模块边界、API flow、前后端关系、数据流等。

#### 4.1.3 输出
一个结构化的上下文对象，必须包含以下字段：

| 输出字段 | 说明 |
|---------|------|
| 相关模块 | 与任务相关的模块列表，每个模块含名称和一句话职责描述。 |
| 关键文件 | 需要 AI Coder 阅读或修改的文件路径列表，含文件类型标记（前端/后端/配置/测试）。 |
| 可复用能力 | 项目中已有的、可被本次任务复用的公开接口或函数，含调用指引。 |
| 禁止修改的核心 | 本次任务可能接近的受保护核心文件，必须明确提醒不可修改。 |
| 风险点 | 本次任务容易踩的坑，例如不要改 GA4 event name、不要绕过 auth guard 等。 |
| 建议测试 | 与本次改动相关的测试文件或需要新增的测试场景。 |

上下文总长度必须控制在 **1500–3000 token** 之间。

#### 4.1.4 核心机制
- **知识图谱构建**：将代码库扫描为图结构，节点类型包括模块、文件、函数、类、API 端点、数据库表、事件、配置项；边类型包括归属（belongs_to）、调用（calls）、依赖（depends_on）、数据流（data_flow_to）、测试覆盖（tested_by）。
- **语义路由**：根据任务描述的自然语言语义，在知识图谱中进行相似度匹配和最短路径搜索，找出与任务最相关的子图。
- **上下文压缩**：对子图进行剪枝和摘要，只保留与任务直接相关的节点和边，生成最小上下文。

#### 4.1.5 边界情况
- 如果任务描述过于模糊，系统应要求用户补充，而不是返回不相关的上下文。
- 如果项目结构发生变化（新增/删除/重命名文件），知识图谱必须在下次 `repoctx scan` 时更新。
- 如果任务涉及跨多个模块的改动，上下文应明确标记跨模块风险。

### 4.2 功能 2：Engineering Principle / Structure Guard

#### 4.2.1 目的
防止 AI Coder 为了完成任务而绕过架构原则，确保新代码从诞生第一天就符合规范。

#### 4.2.2 输入
- 当前变更集中的所有新增或修改的文件。
- 项目预定义的架构规则（文件结构规范、模块职责定义、命名约定等）。
- 工程宪法（EC-1 至 EC-7）。

#### 4.2.3 输出
结构化的审查报告，每条审查项包含：

| 输出字段 | 说明 |
|---------|------|
| 审查类型 | 结构问题 / 原则违反 / 命名问题 / 重复实现 / 硬编码风险 |
| 涉及文件 | 具体文件路径 |
| 问题描述 | 一句话说明问题 |
| 严重程度 | blocker / warning / suggestion |
| 建议操作 | 具体的修改建议，例如移动文件、拆分函数、使用已有能力等 |
| 违反的宪法条款 | 关联的工程宪法编号（EC-1 至 EC-7） |

#### 4.2.4 核心机制
- **规则匹配**：将新增/修改的代码与预定义的结构规则进行模式匹配。
- **职责分析**：分析新文件或新函数的业务语义，判断是否被放在了错误的目录或模块中。
- **重复检测**：检测新代码是否与已有能力高度相似，提醒复用而非重写。
- **实时塑形**：在 AI Coder 编写代码的过程中持续检查，而非仅在 commit 时一次性检查。

#### 4.2.5 检查清单（非穷举）

系统必须能够检测但不限于以下问题：

- Service 是否职责单一；
- View / Controller / Component 是否被塞入了业务逻辑；
- Utils 目录是否被滥用（是否放入了 domain-specific 逻辑）；
- Common 目录是否被乱塞；
- 前端是否绕过 API client 直接调用后端；
- Django app 是否放错了层级；
- 新文件是否应归属已有模块而非新建独立目录；
- 是否重复实现了已有能力；
- 是否引入了硬编码的临时逻辑；
- 是否引入了未来屎山风险（例如过长的函数、过深的嵌套、魔法数字、无注释的复杂逻辑）。

### 4.3 功能 3：Test Impact & Regression Guard

#### 4.3.1 目的
确保任何行为变化都有测试保护，防止改了逻辑但不补测试、改了 API 但不检查契约、旧 case 被 break 而无人知晓。

#### 4.3.2 输入
- 当前变更集。
- 知识图谱中的测试映射关系（哪些测试覆盖了哪些模块/文件/函数）。
- 项目现有的测试目录和测试文件列表。

#### 4.3.3 输出

| 输出字段 | 说明 |
|---------|------|
| 建议运行的测试 | 基于变更影响分析，列出应运行的测试用例或测试文件。 |
| 缺失的测试 | 识别出变更引入的新行为或边界条件，但现有测试未覆盖的场景。 |
| 可能受影响的旧 case | 基于调用链分析，列出可能因本次变更而失败的已有测试。 |
| 测试缺口严重程度 | blocker（必须补）/ warning（强烈建议补）/ suggestion（最好补） |

#### 4.3.4 核心机制
- **影响范围分析**：通过知识图谱追踪变更文件的上游调用者和下游被调用者，确定影响半径。
- **测试映射匹配**：将影响范围内的函数/模块映射到对应的测试文件和测试函数。
- **缺口识别**：对比变更引入的新逻辑路径与现有测试覆盖的路径，找出未覆盖的分支。
- **前后端一致性**：前端改动需检查对应的 unit test 和 E2E test；后端改动需检查对应的 service test、API contract test 和 integration test。

### 4.4 功能 3.5：Legacy Core Guardian

#### 4.4.1 目的
保护老项目底层代码这一生产资产，防止 AI Coder 为了局部需求顺手修改底层函数、公共服务、API 契约等。

#### 4.4.2 核心原则
> 老代码底层资产默认不可改。可以读，可以理解，可以复用 public interface。但不能为了局部需求乱改内部实现。

#### 4.4.3 输入
- 当前变更集。
- 受保护核心索引（Protected Core Index）。
- 可复用能力索引（Reusable Capability Index）。

#### 4.4.4 输出

**情况 A：变更触碰了受保护核心**

输出必须是一个阻止性的审查报告（blocker），包含：

| 输出字段 | 说明 |
|---------|------|
| 被修改的受保护核心 | 文件路径列表 |
| 核心用途 | 该文件被哪些业务流使用（例如被 free_call、subscription、verification_channel 使用） |
| 阻止原因 | 说明为什么该文件属于受保护核心 |
| 强制要求清单 | 必须逐一满足的条件，否则 commit 被 block：① 解释为什么必须修改核心；② 证明 wrapper/adapter 不够；③ 列出所有受影响的业务流；④ 添加回归测试；⑤ 提供回滚方案。 |

**情况 B：任务涉及的能力已有现成实现**

输出是一个引导性提醒（suggestion 或 warning），包含：

| 输出字段 | 说明 |
|---------|------|
| 用户意图 | 系统推测用户想要实现什么能力 |
| 已有能力 | 项目中已存在的、可满足该意图的公开接口 |
| 复用指引 | 如何调用该接口 |
| 扩展建议 | 如果现有接口不完全匹配，建议在何处包一层 wrapper，而不是修改底层实现 |

#### 4.4.5 核心机制
- **受保护核心索引维护**：由项目维护者手动标记或通过历史稳定性分析自动生成，记录哪些文件/模块/函数属于受保护核心。
- **可复用能力索引维护**：总结已有核心能力的使用方式，形成可检索的能力目录。
- **变更拦截**：在 commit gate 阶段，对变更集中的文件路径与受保护核心索引进行交集检测。
- **语义匹配**：对新增代码的业务意图与可复用能力索引进行语义匹配，推荐复用。

### 4.5 功能 4：Experiment Intelligence Agent

#### 4.5.1 目的
管理长周期实验（模型训练、仿真、NDN 实验等）的结构、运行、监控、总结、异常诊断和下一步建议。

#### 4.5.2 输入
- 实验定义（名称、命令、环境配置、预期输出、预期指标）。
- 实验运行过程中的实时输出（stdout/stderr、日志文件）。
- 实验结束后的结果文件（模型权重、指标日志、可视化图表等）。
- 历史实验记忆（同一实验系列的前几次运行记录）。

#### 4.5.3 输出

**实验运行阶段输出：**

| 输出字段 | 说明 |
|---------|------|
| 预检报告 | 环境、依赖、配置的正确性检查结果 |
| 运行时状态 | 存活状态、运行时长、资源占用 |
| 异常告警 | 运行中断、超时、资源耗尽、指标异常等 |

**实验结束后输出（Post-Run Summary）：**

| 输出字段 | 说明 |
|---------|------|
| 实验摘要 | 实验名称、运行时长、使用的配置、执行状态（成功/失败/中断） |
| 指标变化 | 关键指标与预期对比、与历史最佳对比 |
| 异常识别 | 运行过程中出现的错误、警告、指标突变点 |
| 双轨诊断 | Track A（科学解释）和 Track B（实现怀疑）两条路径的分析 |
| 下一步建议 | 基于诊断结果的建议（调参、改实现、改理论、增加数据等） |
| 实验记忆写入 | 将本次实验完整记录沉淀到 experiment memory |
| 通知 | 通过 Slack / Email 发送实验完成通知和摘要 |

#### 4.5.4 核心机制

**实验结构保护**
- 强制实验目录结构，防止 AI Coder 在实验过程中随意创建、移动、删除文件。
- 实验配置必须版本化，每次运行的配置必须可追踪。

**环境 Setup Memory**
- 记录实验所需的环境变量、依赖版本、硬件要求。
- 如果环境配置发生变化，必须明确标记。

**预检（Preflight Check）**
- 检查实验目录结构是否符合规范；
- 检查必要的输入数据是否存在；
- 检查依赖是否安装且版本匹配；
- 检查硬件资源（GPU、内存、磁盘）是否充足。

**运行时监控（Runtime Monitor）**
- 监控进程存活状态；
- 监控资源占用（CPU、内存、GPU、磁盘 I/O）；
- 监控关键指标日志的产出频率（例如训练 loss 是否按时打印）；
- 超时自动告警和中断建议。

**运行后总结（Post-Run Summary）**
- 自动解析实验输出日志，提取关键指标；
- 对比预期指标和实际指标，识别偏差；
- 如果实验失败或结果异常，触发双轨诊断。

**双轨诊断（Dual-Track Diagnosis）**

这是功能 4 最重要的机制。当实验结果违反预期时，系统必须同时输出两条分析路径，不能只做单一路径的解释：

- **Track A — 科学解释（Scientific Interpretation）**：假设代码实现完全正确，从理论、数据、模型假设的角度解释为什么结果可能合理。例如：数据分布偏移、假设不成立、参数选择不当。
- **Track B — 实现怀疑（Implementation Suspicion）**：假设理论基础正确，从代码实现的角度分析哪里可能出了问题。例如：fusion 逻辑是否 order-dependent、是否错误地 sequential update 了 belief、是否 candidate order 影响了结果、是否数值精度出了问题。

系统必须明确指出哪些检查可以帮助区分 Track A 和 Track B。例如：
- shuffle candidate order 后结果是否变化（如果变化，则 Track B 可能性高）；
- 用 synthetic data 验证（如果 synthetic data 上也不行，则 Track A 可能性高）。

**失败模式记忆（Failure Mode Memory）**
- 记录每次实验失败的原因和诊断结论。
- 在新实验遇到相似问题时，主动引用历史失败经验。

**下一步生成器（Next-Step Generator）**
- 基于双轨诊断的结论，生成具体的下一步实验建议。
- 建议必须可执行、可量化、可验证。

## 5. CLI 命令规范（MVP 定义）

MVP 阶段提供 6 个 CLI 命令。以下对每个命令的用途、输入参数、输出格式和行为进行精确定义。

### 5.1 `repoctx scan`

**用途**：扫描项目，生成知识图谱基础文件。

**输入参数**：
- 无必需参数。可选参数包括项目根目录路径（默认当前目录）。

**行为定义**：
1. 扫描项目文件树，识别模块边界；
2. 识别文件归属关系；
3. 提取 API flow、前后端关系、核心数据流；
4. 生成或更新 `.repograph/` 目录下的索引文件；
5. 如果 `protected_core.yaml` 和 `reusable_capabilities.yaml` 不存在，生成模板文件并提示用户手动填写。

**输出产物**：
- `.repograph/index.json`：项目级索引；
- `.repograph/modules/`：模块级索引目录；
- `.repograph/entities/`：实体级索引目录；
- `.repograph/edges/`：关系级索引目录；
- `.repograph/rules/`：规则目录；
- `.repograph/protected_core.yaml`：受保护核心索引；
- `.repograph/reusable_capabilities.yaml`：可复用能力索引。

### 5.2 `repoctx context "<task>"`

**用途**：根据任务描述生成 AI Coder 所需的最小上下文。

**输入参数**：
- 必需：任务描述字符串（自然语言）。
- 可选：上下文长度限制（默认 3000 token，最低 1500 token）。

**行为定义**：
1. 加载知识图谱索引；
2. 对任务描述进行语义解析；
3. 在知识图谱中匹配相关子图；
4. 查询受保护核心索引和可复用能力索引；
5. 生成结构化上下文报告并输出到终端。

**输出格式**：必须包含 4.1.3 节定义的全部字段。

### 5.3 `repoctx status`

**用途**：持续查看当前 working tree 的健康度。

**输入参数**：无。

**行为定义**：
1. 统计当前变更集的文件数和行数；
2. 识别变更涉及的模块数量；
3. 判断变更是否跨前后端；
4. 判断变更是否包含测试文件；
5. 检查变更是否与受保护核心索引有交集；
6. 综合以上信息给出健康度评估和 commit 建议。

**输出字段**：
- 改动文件数、改动行数（新增/删除）；
- 涉及模块列表；
- 是否跨前后端（是/否）；
- 是否修改了测试文件（是/否）；
- 是否触碰受保护核心（是/否，如果是则列出文件）；
- 健康度评级（green / yellow / red）；
- commit 建议（建议提交 / 建议拆分 / 建议补充测试）。

### 5.4 `repoctx commit-check`

**用途**：Commit 前的统一 Gate 检查。

**输入参数**：无。

**行为定义**：
1. 汇总当前变更集；
2. 触发功能 2（Structure Guard）检查；
3. 触发功能 3（Regression Guard）检查；
4. 触发功能 3.5（Legacy Core Guardian）检查；
5. 汇总所有检查结果，给出通过/阻止结论。

**输出字段**：
- 总体结论：pass / block；
- 功能 2 审查报告列表；
- 功能 3 审查报告列表；
- 功能 3.5 审查报告列表；
- 如果结论为 block，必须明确列出所有 blocker 项和必须完成的操作。

### 5.5 `repoctx test-impact`

**用途**：专门分析测试影响。

**输入参数**：无（基于当前变更集分析）。

**行为定义**：
1. 分析变更集的影响范围；
2. 映射到相关测试；
3. 识别测试缺口；
4. 识别可能受影响的旧 case。

**输出字段**：
- 建议运行的测试列表（含测试文件路径和测试函数名）；
- 缺失的测试场景列表（含严重程度和建议）；
- 可能受影响的旧 case 列表（含关联的已有测试）。

### 5.6 `repoctx exp run` 与 `repoctx exp summarize`

**用途**：实验的运行管理和总结。

#### `repoctx exp run`

**输入参数**：
- `--name`（必需）：实验名称，全局唯一；
- `--cmd`（必需）：实验运行的命令；
- `--config`（可选）：实验配置文件路径；
- `--notify`（可选）：实验完成后通知的 Slack channel 或 Email 地址。

**行为定义**：
1. 检查实验名称是否已存在（存在则报错）；
2. 执行预检（Preflight Check）；
3. 如果预检失败，输出预检报告并终止；
4. 如果预检通过，开始运行实验命令；
5. 运行时持续监控（进程存活、资源占用、关键指标产出）；
6. 实验结束后自动触发 `repoctx exp summarize`。

#### `repoctx exp summarize`

**输入参数**：
- `--run`（必需）：实验名称。

**行为定义**：
1. 读取实验运行期间的所有输出日志和结果文件；
2. 提取关键指标；
3. 识别异常；
4. 执行双轨诊断（如果结果异常或不符合预期）；
5. 生成 Post-Run Summary；
6. 将实验记忆写入 `.repograph/experiments/`；
7. 发送通知（如果配置了）。

**输出字段**：见 4.5.3 节。

## 6. 数据结构与配置文件规范

MVP 阶段使用 YAML / JSON 作为数据格式，不引入数据库。以下对每个配置文件的字段、类型、约束进行定义。

### 6.1 `.repoctx.yaml`（项目级主配置）

这是项目的 RepoCtx Guard 主配置文件，位于项目根目录。

**必须包含的顶级字段**：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `project_name` | 字符串 | 项目名称 |
| `version` | 字符串 | RepoCtx Guard 配置格式版本 |
| `language` | 字符串 | 项目主语言（影响扫描策略） |
| `framework` | 字符串 | 项目主框架（影响规则模板） |
| `scan_paths` | 字符串数组 | 需要扫描的目录列表 |
| `exclude_paths` | 字符串数组 | 排除扫描的目录列表（如 node_modules, .git, venv） |
| `modules` | 对象数组 | 模块定义列表，每个对象含 `name`、`path`、`type` |
| `rules_file` | 字符串 | 工程规则文件路径 |
| `protected_core_file` | 字符串 | 受保护核心索引文件路径 |
| `reusable_capabilities_file` | 字符串 | 可复用能力索引文件路径 |
| `experiment_dir` | 字符串 | 实验记忆存储目录 |
| `model_provider` | 对象 | 后台模型提供商配置，见 6.7 节 |

### 6.7 后台模型调用规范（腾讯 MaaS）

系统在执行语义分析、上下文路由、双轨诊断等需要大模型推理能力的环节时，需调用外部 LLM API。当前暂定使用 **腾讯混元 MaaS 平台** 提供的模型服务。

#### 6.7.1 API 端点与模型信息

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 服务商 | 腾讯混元 MaaS | Tencent MaaS Platform |
| API 基础地址 | `https://tokenhub.tencentmaas.com/v1` | 统一入口前缀 |
| 聊天补全端点 | `POST /chat/completions` | 标准 OpenAI-compatible 端点 |
| 当前默认模型 | `deepseek-v4-flash-202605` | 暂定使用的模型版本 |
| 协议格式 | OpenAI API-compatible JSON | 请求与响应均遵循 OpenAI API 规范 |
| 认证方式 | HTTP Header Bearer Token | `Authorization: Bearer {api_key}` |

#### 6.7.2 请求格式定义

系统向模型端点发送的请求必须遵循以下结构：

- **HTTP 方法**：`POST`
- **Headers**：
  - `Authorization`: `Bearer {api_key}`，其中 `{api_key}` 从项目配置中读取。
  - `Content-Type`: `application/json`
- **Body**（JSON）：
  - `model`：字符串，必填。当前固定为 `deepseek-v4-flash-202605`，后续可通过配置覆盖。
  - `messages`：数组，必填。标准 OpenAI 消息格式，每条消息包含 `role`（`system` / `user` / `assistant`）和 `content`（字符串）。
  - `stream`：布尔值，可选。默认 `false`（同步模式）。若未来需要流式输出，需单独评估与 CLI 的兼容性。
  - 其他 OpenAI 标准字段（如 `temperature`、`max_tokens`、`top_p`）暂不启用，后续如有调优需求再纳入配置。

#### 6.7.3 响应格式定义

- **同步模式**（`stream: false`）：返回完整 JSON，系统需解析 `choices[0].message.content` 获取模型输出。
- **错误响应**：遵循标准 HTTP 状态码。常见错误包括：
  - `401 Unauthorized`：API Key 无效或已过期。
  - `429 Too Many Requests`：触发速率限制，需实现指数退避重试。
  - `500/502/503`：服务端异常，需标记为临时失败并提示用户。

#### 6.7.4 API Key 存储策略

- **存储位置**：API Key 不硬编码在代码或文档中，必须存储在项目级配置文件内。
- **配置文件**：支持三种来源，按优先级递减：
  1. `.repoctx.yaml` 的 `model_provider.api_key` 字段。
  2. 环境变量 `REPOCTX_TENCENT_API_KEY`。
  3. 项目根目录 `config.ini` 中 `[DEFAULT]` 节的 `tencent_cloud_llm_api_key`。
- **读取优先级**：
  1. 优先读取 `.repoctx.yaml` 中 `model_provider.api_key`。
  2. 若该字段为空，尝试读取环境变量 `REPOCTX_TENCENT_API_KEY`。
  3. 若前两者皆无，尝试读取 `config.ini` 中 `[DEFAULT]` 节的 `tencent_cloud_llm_api_key`。
  4. 若三者皆无，则报错并提示用户配置 API Key。
- **安全约束**：`.repoctx.yaml` 和 `config.ini` 必须被加入 `.gitignore`，禁止提交到版本仓库；系统扫描时应自动检查并警告。

#### 6.7.5 调用场景清单

以下系统功能在实现时必须通过上述模型端点进行推理：

| 功能模块 | 调用目的 | 输入示例 | 输出示例 |
|---------|---------|---------|---------|
| 功能 1：Context Router | 任务语义解析、知识图谱子图匹配 | 用户任务描述 + 模块/文件列表 | 相关模块与关键文件列表 |
| 功能 2：Structure Guard | 新代码语义分析、职责判断、重复检测 | 新增文件内容 + 已有模块描述 | 审查报告（问题+建议） |
| 功能 4：双轨诊断 | Track A 科学解释、Track B 实现怀疑 | 实验结果异常描述 + 代码片段 + 理论假设 | 双轨诊断报告 |

#### 6.7.6 边界与约束

- **网络不可用**：若调用时无网络连接，需缓存用户请求，返回明确错误提示，不得静默失败。
- **超时设置**：单次请求超时上限暂定为 60 秒；超过则标记为超时，建议用户检查网络或稍后重试。
- **Token 限制**：当前模型版本上下文长度需以腾讯官方文档为准；系统发送的 prompt 必须做截断处理，避免超限。
- **模型切换**：`model_provider.model` 字段预留为可配置项，后续如需切换模型版本（如升级到更新的 DeepSeek 版本或其他腾讯混元模型），仅需修改配置，无需改动调用逻辑。

## 7. 核心流程定义

### 7.1 标准研发任务闭环

**流程名称**：Standard Development Task Loop  
**触发条件**：用户通过 AI Coder 提出代码修改任务  
**参与者**：用户、AI Coder、RepoCtx Guard

**步骤定义**：

| 步骤 | 动作 | 执行者 | 输出 |
|------|------|--------|------|
| S1 | 用户向 AI Coder 提出任务 | 用户 | 任务描述 |
| S2 | AI Coder 或用户调用 `repoctx context "<task>"` | AI Coder / 用户 | 结构化上下文 |
| S3 | AI Coder 基于上下文开始修改代码 | AI Coder | 代码变更 |
| S4 | 开发过程中定期调用 `repoctx status` | AI Coder / 用户 | 健康度报告 |
| S5 | 开发过程中 Structure Guard 持续运行（可通过 IDE 插件或文件监听触发） | RepoCtx Guard | 实时结构审查报告 |
| S6 | 准备 commit 前调用 `repoctx test-impact` | AI Coder / 用户 | 测试影响分析 |
| S7 | Commit 前调用 `repoctx commit-check` | AI Coder / 用户 | Gate 检查结果 |
| S8 | 如果 Gate 通过，允许 commit；如果存在 blocker，必须修复后重新检查 | RepoCtx Guard | pass / block |
| S9 | Commit 后结束 | — | — |

### 7.2 受保护核心修改审批流程

**流程名称**：Protected Core Modification Approval  
**触发条件**：变更集与受保护核心索引产生交集  
**审批结果**：approve / reject / request_changes

**必须提供的材料**：

1. **必要性解释**（Required）：为什么必须修改该核心文件，而不是通过 wrapper/adapter 扩展？
2. **替代方案证明**（Required）：证明 wrapper/adapter 确实无法满足需求。
3. **影响流清单**（Required）：列出所有受该修改影响的业务流和模块。
4. **回归测试**（Required）：为修改添加的回归测试清单和预期结果。
5. **回滚方案**（Required）：如果修改上线后出问题，如何快速回滚。

**审批规则**：
- 如果任何一项材料缺失，Commit Gate 直接 block，不允许绕过。
- 如果所有材料齐全但影响流超过 3 个核心业务流，必须额外提供灰度发布计划。
- 审批记录必须写入 `.repograph/protected_core_modification_log.yaml`。

### 7.3 实验管理闭环

**流程名称**：Experiment Management Loop  
**触发条件**：用户通过 `repoctx exp run` 启动实验  
**参与者**：用户、实验进程、RepoCtx Guard

**步骤定义**：

| 步骤 | 动作 | 执行者 | 输出 |
|------|------|--------|------|
| E1 | 用户定义实验（名称、命令、配置） | 用户 | 实验定义 |
| E2 | 用户调用 `repoctx exp run --name <name> --cmd <cmd>` | 用户 | 运行请求 |
| E3 | 系统执行预检 | RepoCtx Guard | 预检报告 |
| E4 | 如果预检失败，终止并输出报告 | RepoCtx Guard | 失败报告 |
| E5 | 如果预检通过，启动实验进程 | RepoCtx Guard | 进程 PID |
| E6 | 运行时持续监控 | RepoCtx Guard | 运行时状态 |
| E7 | 实验进程结束（成功/失败/中断） | 实验进程 | 退出码 |
| E8 | 系统自动触发 `repoctx exp summarize --run <name>` | RepoCtx Guard | Post-Run Summary |
| E9 | 如果结果异常，执行双轨诊断 | RepoCtx Guard | 双轨诊断报告 |
| E10 | 将实验记忆写入 `.repograph/experiments/` | RepoCtx Guard | 实验记录文件 |
| E11 | 发送通知 | RepoCtx Guard | Slack / Email 通知 |
| E12 | 用户基于摘要和诊断决定下一步 | 用户 | 下一步行动 |

## 8. 异常处理与边界情况

### 8.1 知识图谱相关

| 异常情况 | 处理策略 |
|---------|---------|
| 扫描时遇到无法解析的文件格式 | 跳过该文件，记录警告日志，不影响整体扫描。 |
| 知识图谱索引文件损坏 | 启动时检测到 JSON 格式错误，提示用户运行 `repoctx scan` 重建索引。 |
| 任务描述过于模糊 | 拒绝生成上下文，返回澄清问题列表，要求用户补充。 |
| 任务涉及未识别的模块 | 在上下文中标记 "未知模块"，提示用户检查项目结构或补充模块定义。 |

### 8.2 审查与 Gate 相关

| 异常情况 | 处理策略 |
|---------|---------|
| 变更集为空 | `repoctx status` 和 `repoctx commit-check` 输出 "无变更" 并正常退出。 |
| 规则文件缺失 | 扫描时自动生成规则模板文件，并提示用户手动补充。所有规则检查默认通过，但输出警告。 |
| 受保护核心索引为空 | 功能 3.5 的保护机制不生效，输出警告提示用户维护索引。 |
| 测试目录不存在 | 功能 3 跳过测试影响分析，输出警告提示用户项目缺少测试。 |

### 8.3 实验相关

| 异常情况 | 处理策略 |
|---------|---------|
| 实验名称重复 | `repoctx exp run` 报错，要求用户更换名称或先删除旧实验记录。 |
| 实验进程崩溃 | 运行时监控检测到进程退出，立即标记实验状态为 failed，并触发 summarize。 |
| 实验超时 | 运行时监控检测到超过预设超时时间，发送告警，由用户决定是否中断。 |
| 实验结果文件缺失 | summarize 阶段检测到预期结果文件不存在，在摘要中标记 "结果不完整"，并尝试基于已有日志生成部分摘要。 |
| 双轨诊断无法区分 Track A/B | 明确在报告中说明 "当前信息不足以区分两条路径"，并列出可以补充的验证实验。 |

## 9. 验收标准

以下标准用于判断 RepoCtx Guard 的 MVP 实现是否合格。

### 9.1 功能验收

| 功能 | 验收标准 |
|------|---------|
| `repoctx scan` | 能在 3 分钟内完成一个中等规模项目（约 500 个文件）的扫描，生成所有索引文件且不报错。 |
| `repoctx context` | 输入 10 个不同的任务描述，80% 以上能返回包含正确模块和关键文件的上下文。 |
| `repoctx status` | 能准确识别变更集的文件数、行数、涉及模块、是否跨前后端、是否触碰受保护核心。 |
| `repoctx commit-check` | 能正确汇总 Structure Guard、Regression Guard 和 Legacy Core Guardian 的检查结果，blocker 项必须被阻止。 |
| `repoctx test-impact` | 能准确映射变更到相关测试，并识别出明显的测试缺口。 |
| `repoctx exp run / summarize` | 能完整执行预检、运行、监控、总结流程，实验结束后生成符合格式要求的摘要。 |

### 9.2 性能验收

| 指标 | 目标值 |
|------|--------|
| 项目扫描时间 | 500 个文件的项目，扫描不超过 180 秒 |
| 上下文生成时间 | 不超过 5 秒 |
| commit-check 时间 | 不超过 10 秒 |
| 运行时监控间隔 | 不超过 30 秒 |

### 9.3 可用性验收

| 指标 | 目标值 |
|------|--------|
| 配置文件模板生成 | 首次扫描后自动生成所有模板文件 |
| 错误提示清晰度 | 所有错误和警告必须包含 "问题是什么" 和 "建议怎么做" |
| 输出可读性 | CLI 输出必须结构化、带颜色区分 severity、支持 `--format json` |

## 10. 后续迭代方向（非 MVP）

以下功能不在 MVP 范围内，但应在架构设计时预留扩展点：

1. **IDE 插件**：VS Code / Cursor / JetBrains 插件，实现实时 Structure Guard 和 inline 提醒。
2. **Web Dashboard**：可视化知识图谱、审查历史、实验管理界面。
3. **数据库后端**：当知识图谱规模过大时，从文件系统迁移到图数据库（如 Neo4j）。
4. **CI/CD 集成**：GitHub Actions / GitLab CI 插件，在 PR 阶段自动运行 commit-check。
5. **自然语言规则**：允许用户用自然语言描述规则，系统自动转换为可执行检查。
6. **多语言支持**：除了 Python / JavaScript / Vue / Django，扩展到 Go、Rust、Java 等。
7. **实验自动调参**：基于历史实验记忆，自动建议下一组超参数。
8. **团队协作**：多人项目中的变更冲突检测和审查记录共享。
